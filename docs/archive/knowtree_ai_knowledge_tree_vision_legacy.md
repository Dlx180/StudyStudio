# KnowTree：面向个性化学习的 AI 知识树系统

## 1. 项目理念

KnowTree 的目标是开发一个由 AI 驱动的个性化学习软件，帮助用户从零散的学习资料中建立清晰、可追踪、可成长的知识结构。

在真实学习过程中，学习困难往往不只来自某个知识点本身，而是来自更深层的问题：用户缺乏对整个学科结构的把握，不清楚不同概念之间的依赖关系，也很难判断自己到底缺少哪些前置知识。同一本书、同一门课程、同一个领域，对于不同用户来说难度并不相同，因为每个人的学习背景、目标、已有知识和理解方式都不同。

KnowTree 希望解决的核心问题是：

> 将用户的学习资料、学习目标和个人状态组织成一张可演化的知识网络，并以清晰的知识树或阅读树形式呈现给用户。

传统学习工具通常以“文件”“课程”“笔记”或“问答”为中心。KnowTree 的核心理念则是以“知识关系”为中心：文件是知识的来源，阅读树是学习的路径，知识网是长期沉淀，用户状态则记录个人在不同学习目标下的掌握情况。

软件最终希望实现的不只是“总结文件”或“AI 答疑”，而是构建一个完整的学习系统：

- 帮助用户管理本地和网络学习资料；
- 从 PDF、PPT、网页、视频、论文、GitHub 仓库等资源中提取学习结构；
- 将不同风格、不同粒度的学习材料转化为适合用户阅读的知识树；
- 记录用户在首次学习、复习、考试准备、项目实践等不同场景中的学习状态；
- 逐步沉淀个人知识网络；
- 支持后续的资源推荐、学习路径生成、知识树共享和社区协作。

## 2. 核心思想：底层是网，展示是树

KnowTree 的知识结构采用“底层知识网，前端知识树”的设计。

知识本身通常不是严格树状的。一个概念可能属于多个领域，也可能在不同课程、不同教材、不同项目中反复出现。例如“特征值”既可以出现在高等代数中，也可以出现在机器学习、图论、物理和信号处理中。因此，底层数据结构更适合使用知识网，用节点和边表达概念、资源、学习单元、前置关系、相似关系、引用关系和应用关系。

但对于用户来说，直接面对复杂知识图谱并不适合学习。用户需要的是在某个具体目标下清晰、有顺序、可执行的学习路径。因此，KnowTree 会根据用户目标、资料范围和学习状态，从底层知识网中生成不同用途的树状视图。

同一张知识网可以生成多种树：

- 课堂同步阅读树；
- 初次学习知识树；
- 考试复习树；
- 项目实践树；
- 补基础路径树；
- 论文阅读树；
- 快速入门树。

因此，KnowTree 中的“树”不是固定不变的目录，而是从知识网中投影出来的学习视图。底层知识网负责长期沉淀，前端树结构负责当前任务下的学习导航。

## 3. 学习资料管理器：文件是入口，知识是索引

KnowTree 的主体形态可以理解为一个面向学习的资料管理器。但它不是传统文件管理器，也不是简单的网盘或笔记软件。

传统文件管理器以文件夹和文件路径为中心：

```text
文件夹 → 文件 → 打开阅读
```

KnowTree 以知识关系为中心：

```text
学习目标 → 阅读树 / 知识树 → 相关文件页码与资源片段 → AI 解释与练习
```

用户上传的 PDF、PPT、教材、论文、课堂讲义、作业、网页、视频和代码仓库，都会被视为学习资源。系统会识别资源结构，将其转化为可学习的 ReadingUnit，并进一步组织成 UnitTree 或 KnowledgeTree。

在这个系统中，文件不再只是静态资料，而是知识网络中的证据来源。用户点击某个知识节点时，可以看到它来自哪些文件、哪些页、哪些 PPT、哪些视频片段或哪些网页内容。这样既保留了原始材料的可信度，又能借助 AI 对材料进行结构化重组。

## 4. 关键对象

KnowTree 的核心对象包括：

### Resource

Resource 表示原始学习资料，例如 PDF、PPT、网页、视频、论文、GitHub 仓库或用户笔记。它是系统的输入来源。

### FileTree

FileTree 表示资料自身的结构，例如 PDF 目录、章节、小节、页码、PPT 顺序、slide 标题等。FileTree 反映的是作者原本如何组织内容。

### Page / Slide

Page 或 Slide 是最稳定的定位单位。对于 PDF 和 PPT，页码或幻灯片编号是用户最容易理解、也最容易回溯到原文的位置。系统不需要对每一页都建立复杂知识属性，但需要保留页级定位能力。

### ReadingUnit

ReadingUnit 是 KnowTree 第一阶段最核心的学习单位。它通常由连续若干页或若干张 slide 构成，表示一个适合用户学习的内容单元。

例如：

```text
Lecture 3 第 6-10 页：梯度下降的直观理解
Chapter 2 第 21-35 页：线性回归模型
PPT 第 12-18 页：损失函数与优化目标
```

ReadingUnit 不是简单按页数切分，而是根据标题、语义连续性、学习目标、用户需求和文件结构综合生成。

### UnitTree

UnitTree 是由 ReadingUnit 构成的阅读树或学习树。它将原始资料转化为更适合学习的结构，帮助用户理解“当前资料在讲什么”“我读到了哪里”“这些内容之间是什么关系”。

### KnowledgeNode

KnowledgeNode 表示长期沉淀的抽象知识点，例如“梯度下降”“学习率”“过拟合”“特征值”“贝叶斯公式”等。ReadingUnit 可以映射到一个或多个 KnowledgeNode。

### KnowledgeGraph

KnowledgeGraph 是长期沉淀的知识网络，记录知识点之间的前置关系、包含关系、相似关系、应用关系和易混关系。第一阶段不需要构建完整知识图谱，但数据结构需要为后续扩展保留空间。

### StateOverlay

StateOverlay 记录用户在不同用途树上的学习状态。它不直接污染底层知识网，而是覆盖在具体学习场景上。

同一个知识点在不同用途下可能有不同状态：

- 首次学习：未学，需要详细解释；
- 复习：已学过，需要快速回顾；
- 考试准备：需要练习题和易错点；
- 项目实践：需要代码示例和应用方式。

因此，状态层应该跟随具体学习树、学习模式和学习会话变化，而不是写死在知识点本身。

## 5. 核心能力：UnitReading

UnitReading 是 KnowTree 的底层核心接口之一。它负责将某个资料范围转化为适合学习的 UnitTree。

它的输入可以是：

- 整份 PPT；
- PDF 的某一章；
- 某几页教材；
- 多个课件组成的一个主题；
- 网页内容；
- 视频字幕片段；
- 用户选中的资料范围。

它的输出是：

- ReadingUnit 列表；
- UnitTree 结构；
- 每个 Unit 的标题、页码范围、摘要、学习目标、主要概念和难度；
- Unit 之间的顺序、前置关系和层级关系；
- 与原始文件页码或资源片段的对应关系。

这个接口的意义在于统一不同来源的学习资料。无论输入是 PDF、PPT、网页、视频还是 GitHub 项目，系统最终都可以通过 UnitReading 生成可学习、可导航、可追踪的 UnitTree。

## 6. 用户使用场景

KnowTree 的典型使用方式是：

1. 用户创建一个学习空间，例如“机器学习课程”“凸优化复习”“论文阅读计划”。
2. 用户上传 PDF、PPT、教材章节、作业或网页链接。
3. 系统识别资料类型，生成 FileTree 和基础索引。
4. 系统根据资料范围调用 UnitReading，生成 UnitTree。
5. 用户在左侧阅读原始 PDF/PPT，在右侧查看对应阅读树。
6. 用户点击右侧树节点时，左侧跳转到对应页码。
7. 用户滚动阅读时，右侧树自动高亮当前 ReadingUnit。
8. 用户可以对当前 Unit、当前页或选中内容提问。
9. 用户可以标记理解程度、收藏节点、生成练习、记录问题。
10. 系统逐渐沉淀用户的个人知识网和学习状态。

第一阶段的核心界面可以设计为：

```text
左侧：PDF / PPT 阅读区
右侧：UnitTree / 当前 Unit 摘要 / AI 问答 / 学习状态
```

这使 KnowTree 不脱离用户原始资料，而是在可信材料旁边提供结构化导航和 AI 学习辅助。

## 7. 完整软件形态

KnowTree 的完整形态不仅是一个 PDF/PPT 阅读器，而是一个围绕知识网展开的学习操作系统。

完整软件应包括以下能力：

### 7.1 学习资料管理

支持管理本地和网络学习资料，包括：

- PDF；
- PPT/PPTX；
- Word / Markdown；
- 网页；
- 视频；
- 论文；
- GitHub 仓库；
- 作业；
- 用户笔记；
- 图片或扫描件。

资料可以按照课程、项目、主题、考试、研究方向组织。系统会为每份资料建立结构索引，并将其挂接到知识网中。

### 7.2 阅读树与知识树生成

系统可以根据资料生成阅读树，也可以根据用户目标和已有知识生成知识树。

阅读树更接近原始资料结构，用于辅助阅读。知识树更接近学习目标结构，用于规划路径、补充前置知识和复习。

### 7.3 个人知识网沉淀

随着用户不断学习，系统会将不同资料中的 ReadingUnit 和知识点逐渐合并到个人知识网中。

例如用户在“线性代数”中学过“特征值”，后来在“机器学习”中学习 PCA 时，系统可以发现两者关联，并提示用户复习相关前置知识。

### 7.4 学习状态管理

系统记录用户在不同用途树上的状态，包括：

- 是否读过；
- 是否理解；
- 是否掌握；
- 是否需要复习；
- 是否有疑问；
- 是否做过练习；
- 是否在考试或项目中使用过。

状态不是单一进度条，而是和学习目标、学习模式、资料范围、知识节点共同关联。

### 7.5 AI 辅助学习

AI 能力包括：

- 资料总结；
- UnitTree 生成；
- 当前页解释；
- 选区问答；
- 公式和图表解释；
- 前置知识提示；
- 练习题生成；
- 易错点总结；
- 不同难度解释；
- 学习路径重组。

AI 不应该替代原始资料，而应该帮助用户理解、组织和导航资料。

### 7.6 网络资料插件

后续可以开发浏览器插件，用于采集网页、视频、论文、GitHub 项目等网络资源。

插件功能包括：

- 一键加入 KnowTree；
- 将网页加入某个 workspace；
- 提取视频字幕并生成 UnitTree；
- 解析 GitHub README 和项目结构；
- 将在线 PDF 或课程页面加入学习资料库。

插件不是主产品，而是资料入口。主学习场景仍然在 KnowTree Web App 中完成。

### 7.7 共享知识树与公共知识图谱

长期来看，KnowTree 可以支持共享知识树和公共知识图谱。

用户可以选择发布自己的学习路径、资源标注、课程树或知识树模板。公共图谱可以作为参考骨架，而不是直接覆盖用户的个人图谱。用户可以导入、复制、修改、扩展公共知识树，形成自己的学习路径。

未来可以进一步支持：

- 知识树模板市场；
- 课程共建；
- 学习小组；
- 打卡和互相监督；
- 节点评论；
- 资源推荐；
- 社区共创知识图谱。

## 8. 技术路线

KnowTree 推荐采用 Web App 作为主产品形态。

第一阶段应优先实现：

- PDF 上传；
- PDF 阅读器；
- 左侧阅读、右侧 UnitTree；
- 文件页码索引；
- UnitReading 接口；
- ReadingUnit 数据结构；
- UnitTree 展示；
- Unit 状态记录；
- 当前 Unit 问答。

PPT 可以通过后端转换为 PDF 后复用同一阅读器。这样可以避免前端同时维护 PDF 和 PPT 两套阅读系统。

推荐技术路线包括：

- 前端：Next.js / React；
- PDF 阅读：PDF.js 或其 React 封装；
- 后端：FastAPI / Python；
- 数据库：PostgreSQL；
- 向量检索：pgvector；
- 任务队列：Redis + Celery / RQ；
- 文件存储：S3 / MinIO 或本地对象存储；
- 文档解析：PyMuPDF、pdfplumber、LibreOffice headless、unstructured；
- AI 层：大模型 API + embedding + prompt version 管理。

大模型不应该负责所有文件处理。普通程序负责文件上传、渲染、文本抽取、页码管理、目录识别、缓存和状态记录；大模型主要负责学习结构重组、Unit 命名、摘要、前置知识判断、问答和解释。

## 9. 开发计划

KnowTree 应采用分阶段开发，而不是一次性构建完整平台。

### 阶段一：核心阅读工作台

目标是实现最小可用闭环：

```text
上传 PDF → 解析页码 → 生成 UnitTree → 左侧阅读 → 右侧导航 → 当前 Unit 问答 → 状态记录
```

这一阶段重点验证：AI 生成的 UnitTree 是否真的能帮助用户阅读和学习资料。

### 阶段二：UnitTree 编辑与重生成

AI 生成的 UnitTree 一定会有不准确之处，因此需要提供用户编辑能力：

- 修改 Unit 标题；
- 调整页码范围；
- 合并 Unit；
- 拆分 Unit；
- 删除错误 Unit；
- 重新生成局部 Unit；
- 保存用户确认版本。

这一阶段重点是让 AI 结果可修正、可维护，而不是一次生成后不可改变。

### 阶段三：PPT 支持

支持 PPT/PPTX 上传，并在后端转换为 PDF 进行阅读。系统需要保留 slide 到 page 的映射，确保用户仍能按幻灯片定位。

这一阶段可以覆盖大量课堂资料场景。

### 阶段四：多文件 workspace 与 UnitTree 合并

支持一个学习空间中包含多份资料，例如一门课的多个 lecture、作业和教材章节。系统需要能够合并多个 UnitTree，识别重复内容、前置内容和跨文件主题。

这一阶段开始形成课程级学习树。

### 阶段五：轻量知识网沉淀

在 UnitTree 稳定后，逐步引入 KnowledgeNode 和 KnowledgeGraph。系统从 ReadingUnit 中提取稳定知识点，并建立知识点之间的前置关系、包含关系和相关关系。

这一阶段的目标不是构建完整知识图谱，而是让知识点开始跨文件、跨课程复用。

### 阶段六：学习状态与复习系统

引入更完整的 StateOverlay，根据用户在不同学习模式下的状态生成不同用途树：

- 第一次学习树；
- 复习树；
- 考试冲刺树；
- 项目实践树；
- 补基础树。

这一阶段开始真正体现个性化学习。

### 阶段七：浏览器插件与网络资源

开发浏览器插件，让用户可以将网页、论文、视频、GitHub 仓库加入 KnowTree。网络资源进入系统后，也通过 UnitReading 转化为 UnitTree，并与个人知识网连接。

### 阶段八：公共知识树与协作

在个人学习系统稳定后，引入共享和协作能力：

- 发布知识树模板；
- 导入公共学习路径；
- 多用户共同维护资源节点；
- 共享课程树；
- 社区补充资源；
- 学习小组和打卡。

这一阶段将 KnowTree 从个人学习工具扩展为学习社区和知识结构平台。

## 11. 项目愿景

KnowTree 的长期愿景是成为一个面向学生、研究者和终身学习者的 AI 学习操作系统。

它不是简单的 AI 聊天工具，也不是普通网盘、笔记软件或 PDF 阅读器。它希望将用户的学习资料、学习目标、知识关系和个人状态连接起来，让学习从“打开一个文件”变成“沿着一棵适合自己的知识树前进”。

最终，KnowTree 应该帮助用户回答这些问题：

- 我现在应该学什么？
- 这个知识点依赖哪些前置知识？
- 我的资料里哪里讲过它？
- 这几份资料之间有什么关系？
- 我是否真正掌握了它？
- 我应该如何复习？
- 我能否把别人的学习路径转化为适合自己的路径？
- 我能否将自己的学习成果沉淀为可复用的知识树？

KnowTree 的核心价值是将 AI 的生成能力、资料管理能力、知识结构能力和个性化学习状态结合起来，构建一个能够长期陪伴用户学习和成长的软件系统。
